### 1. 4:5 인스타그램 피드 최적화 및 입체적 카드 UI 디자인 개편
- **구현 내용**:

  - 카드 내부는 화이트 룩북 레이아웃으로, 외부 카드 테두리는 입체 쉐도우와 `BorderRadius.circular(16)`를 입혀 프리미엄 감성을 극대화하고 바깥쪽은 세련된 라이트 그레이 배경으로 구성하여 매거진 스타일 룩을 완성했습니다.

### 2. 실시간 스마트 코디 추천 하단 상시 노출 복구
- **구현 내용**:
  - 화면 상단 앱바로 은폐되었던 `_buildRecommendationSection`을 캔버스 아래 고정 여백 영역(220px 높이)으로 복원 노출시켰습니다.
  - 추천 리스트의 의류 에셋 카드를 클릭 시 실시간으로 캔버스에 추가되도록 설계하여 직관적인 옷 매칭 편의성을 회복하고, 옷 추가 플로팅 버튼(FAB)의 간섭 예방을 위해 하단 마진을 `285`로 올렸습니다.

### 3. 지능형 가변 배치(Adaptive Layout Mode) 알고리즘 도입
- **구현 내용**:
  - 올려진 의류 에셋들의 총개수와 카테고리를 실시간 집계하여 레이아웃 구도를 가변 분기합니다.
  - 메인 의류가 3벌 이하인 경우 **수직 1열 단일 코디 모드**로 구동되어 X축(50% 중앙선) 칼정렬을 진행하고, 가방 및 아우터는 겹침을 방지하기 위해 각각 좌우 사이드 여백 지대에 분산 배열합니다.
  - 4벌 이상이거나 동일 카테고리 중복 에셋이 존재할 시, 자동으로 2x2 분할 격자 기반의 **다중 격자 콜라주 모드**로 자동 전환하여 겹침 없이 넓게 흩어지도록 정렬합니다.

### 4. 물리적 위젯 레이아웃 제어로 정중앙 정밀 안착 및 겹침 제거
- **구현 내용**:
  - 기존 `Transform.scale` 사용 시 렌더링 영역(RenderBox)의 실제 크기는 150x150으로 변하지 않아 포지션 앵커 포인트 계산이 어긋나던 런타임 버그를 해결했습니다.
  - `Transform.scale` 대신 `Container` 위젯의 실제 `width`와 `height`를 직접 `150.0 * item.scale`로 강제 조절하도록 리팩토링했습니다.
  - 이에 따라 마법봉(🪄) 자동 정렬 시 단일 코디 상의/하의의 가로축 비주얼 중앙점이 캔버스 정중앙에 1px의 오차도 없이 일치하게 정렬되며, 의류 선택 시 생성되는 검은색 사각 테두리도 옷 크기에 맞게 딱 피팅되어 미적 퀄리티를 배가했습니다.
  - 150x150의 투명 터치 마스킹 영역이 사라짐으로써, 드래그 및 터치 조작 반응성도 극대화되었습니다.

### 5. 템플릿 테마 인식형(Template-Aware) 텍스트 겹침 차단 고도화
- **구현 내용**:
  - 배경 테마가 의류와 겹쳐 텍스트 가독성이 훼손되던 문제를 해결하기 위해, 활성 템플릿에 따라 정렬 Y축 오프셋을 수학적으로 조율했습니다.
  - **에디토리얼 상단 소글씨 우회**: Essentials 타이틀 밑에 쌓이는 `STYLE DIARY & ARCHIVE` 영역(약 90px 높이)을 우회하도록, 메인 의류 정렬 Y축을 **`38%` (152px 지점) 이하로 대폭 내려 정렬**하고 상의-하의 간 격차를 넓혀 세로/가로 겹침을 0%로 만들었습니다.
  - **폴라로이드 좁은 영역 내 비겹침 스케일**: 위쪽 12px, 아래쪽 64px 마진으로 세로 가용 영역이 짧은 폴라로이드 모드 안에서 옷이 겹치거나 아래 손글씨 영역을 가리지 않도록, 에셋 스케일을 **`0.50 ~ 0.58` 수준으로 축소하여 비겹침 1열 라인**을 유지시켰습니다.
  - **카탈로그 상단 띠 바 회피**: 좌상단 검은색 띠 헤더 높이만큼 구도를 약 6~8%씩 아래로 끌어내려 배치합니다.

### 6. 소장 옷 개수 기준 카테고리 칩바 동적 정렬
- **구현 내용**:
  - 홈 화면 진입 및 옷장 데이터 로드 완료 시점에 사용자의 옷장에 등록된 의류 개수를 카테고리별로 실시간 집계합니다.
  - 'ALL' 칩은 전체 의류 조회를 위해 항상 첫 번째(맨 왼쪽)에 고정 배치하고, 그 뒤에 오는 서브 카테고리 칩들은 실제 등록된 옷의 벌 수가 많은 순(내림차순)으로 동적 재정렬되도록 구현했습니다.
  - 이를 통해 개개인마다 다른 의류 소장 현황에 최적화된 맞춤형 필터 뷰포트를 제공합니다.

### 7. 개인화 카테고리 설정(Custom Visibility) 기능 및 Preferences 연동
- **구현 내용**:
  - 남성과 여성 사용자 등 선호에 따라 원피스, 치마 등의 특정 카테고리를 노출하고 싶지 않거나 숨기고 싶은 요구사항을 반영했습니다.
  - 가로 스크롤 칩바 맨 우측 끝에 **⚙️ 설정 칩**을 유기적으로 이식하고, 탭 시 10대 카테고리의 노출 상태를 개별 토글할 수 있는 미니멀 바텀 시트를 제공합니다.
  - 사용자가 저장한 activeCategories 설정을 Firestore 사용자 정보 문서에 `activeCategories` 필드로 병합 저장(`SetOptions(merge: true)`)하여 기기를 바꾸더라도 필터 셋업이 동일하게 자동 보존됩니다.

## 2026-06-17 (v5.0) — 운영/개발 환경 분리 마일스톤

### 1. 운영(prod) / 개발(dev) Firebase 프로젝트 완전 분리
**문제**: 단일 Firebase 프로젝트(`digital-closet-32c43`)만 존재해, 로컬 개발이나 신규 기능 테스트가 곧바로 운영 데이터(사용자·옷장·OOTD)에 영향을 줄 위험이 상존함.
**수정 내용**:
- `lib/config/firebase_env.dart` 신설: `--dart-define=ENV=prod|dev` 값에 따라 `FirebaseOptions`를 자동 전환. 기본값은 `dev`로 설정하여 로컬 `flutter run` 시 운영 데이터 접근을 원천 차단.
- `firebase_service.dart`에 하드코딩돼 있던 `FirebaseOptions`/API Key를 제거하고 `FirebaseEnv.options`로 일원화.
- 신규 개발계 프로젝트 `digital-closet-dev` 생성 및 웹 앱(`dress-dev-web`) 등록.
**이유**: 운영 서비스의 안정성을 보장하면서, 개발자가 데이터 오염 걱정 없이 자유롭게 개발·테스트할 수 있는 격리 환경을 확보함.

### 2. 개발계 백엔드 인프라 구성 (Firestore · Storage · Auth)
**문제**: 신규 dev 프로젝트는 빈 상태로, 앱이 요구하는 백엔드 서비스가 미구성 상태였음.
**수정 내용**:
- Firestore 데이터베이스 생성 후 운영계와 동일한 보안 규칙(`firestore.rules`)·복합 색인(`firestore.indexes.json`) 배포.
- Cloud Storage 버킷(`us-west1`) 생성 후 보안 규칙(`storage.rules`) 배포.
- Authentication 이메일/비밀번호 로그인 활성화 (앱은 `createUserWithEmailAndPassword` / `signInWithEmailAndPassword` 사용).
**이유**: 개발계에서도 운영과 동일한 데이터 구조·보안 정책으로 동작하도록 하여, 환경 차이로 인한 버그를 사전에 차단함.

### 3. 환경별 배포 자동화 스크립트
**문제**: 빌드 시 환경 플래그 누락, 잘못된 프로젝트로의 배포 등 휴먼 에러 가능성이 큼.
**수정 내용**:
- `deploy_dev.sh`: `ENV=dev` 웹 빌드 후 `digital-closet-dev` 호스팅에 배포.
- `deploy_prod.sh`: 운영 배포 전 확인 프롬프트(Enter/Ctrl+C)를 거친 뒤 `ENV=prod` 빌드 및 운영 호스팅 배포.
- `.firebaserc`에 `prod`/`dev` 프로젝트 별칭(alias) 등록.
**이유**: 환경 전환과 배포를 표준화하여 실수로 인한 오배포를 방지하고 배포 절차를 단순화함.

### 4. Git 브랜치 전략 수립 (main=개발 / prod=운영)
**문제**: 단일 `main` 브랜치만 존재하여 개발 중인 코드와 운영 배포 코드의 구분이 없음.
**수정 내용**:
- `main` = 개발(dev) 트랙, `prod` = 운영(prod) 트랙으로 이원화하고 원격(origin)에 푸시.
- 워크플로우 확립: `main`에서 개발·검증 → 운영 반영 시 `main → prod` 머지 후 운영 배포.
**이유**: 개발과 운영 릴리스를 명확히 분리하여 배포 안정성과 추적성을 확보함.

### 3. Mac OS NFD(자소분리) 파일명에 따른 웹 에셋 깨짐 해결

**수정 내용**:
- `scripts/rename_to_hex.py` 및 `scripts/normalize_assets.py`를 통해 모든 에셋의 한글 파일명을 유니코드 16진수 hex 형태(영문/숫자)로 강제 일괄 정규화.
- `categories.dart` 내 `getIconPath` 함수를 변경하여 소분류 텍스트를 유니코드 hex로 치환해 1:1 매칭 완료.
**이유**: 크로스 플랫폼 브라우저 환경에서 자소분리로 인한 이미지 누락 버그를 완전히 차단하고, 에셋 렌더링 안정성을 100% 확보함.

### 4. 모바일 물리 뒤로가기 종료 방지 및 UX 개선 (PopScope)
**문제**: 모바일 환경에서 사용자가 뒤로가기 버튼을 누르면 앱이 예고 없이 즉각 종료되어 사용자 경험이 저해됨.
**수정 내용**:
- `main_screen.dart`에 최신 `PopScope` 위젯을 적용하여 뒤로가기 제스처 시 홈(옷장) 탭으로 우선 전환 처리.
- 홈 탭에서는 1차 뒤로가기 시 스낵바로 안전 종료 확인 안내를 제공하고, 2초 이내에 연속 입력 시에만 앱이 종료되도록 이중 방어 로직 구현.
**이유**: 실수로 인한 앱 강제 이탈을 막고 안정적인 모바일 플랫폼 내비게이션 흐름을 확립함.

- **코디 아이디어 화면 소셜 기능 연동**: 이미지 하단에 코디 추천인(`suggestedBy`) 텍스트를 노출하고, 하단에 좋아요/댓글 기능 바를 활성화하여 추천받은 코디 아이디어에도 반응할 수 있도록 개선 (`planned_ootd_detail_screen.dart`).

**이유**: 사용자 피드백을 반영하여 지난 기록들을 보기 쉽게 시각적으로 개선하고, 코디 아이디어 공간까지 마이크로 소셜 경험을 확장함.

### 2. 코디 아이디어 라우팅 오류 수정 및 썸네일 UI 보완
**문제**:

**수정 내용**:
- **라우팅 오류 수정**: `OotdScreen` 내 코디 아이디어 탭 썸네일 클릭 이벤트, 그리고 `CoordinationCanvasScreen` 하단 썸네일 클릭 이벤트를 모두 `PlannedOotdDetailScreen`으로 정상 연결(`Navigator.push`)하도록 수정.
- **추천인 뱃지 오버레이 추가**: 모아보기 바둑판(`OotdScreen`)의 각 썸네일 하단에 반투명한 그라데이션과 함께 `💡 by {추천인}` 텍스트가 표시되도록 Stack 위젯 적용.
- **친구 코디 조회 백엔드 쿼리 수정**: `FirebaseService`의 `getPlannedOOTDPage` 함수에 `targetUserId` 파라미터를 추가하여, 친구 코디를 도와줄 때 해당 친구의 기존 코디 아이디어를 정확히 불러오도록 수정 (`firebase_service.dart`).

**이유**: 코디 아이디어 공간에 확장된 소셜 기능(댓글/좋아요)으로 온전히 진입할 수 있도록 네비게이션 플로우를 재정비하고, 추천인 정보를 직관적으로 표시하여 UX를 향상시킴.

### 3. 친구 OOTD 피드 로딩 성능 최적화

**수정 내용**:

**이유**: 불필요한 네트워크 통신과 이미지 렌더링 부하를 줄이고, 앱의 반응성을 극대화하여 쾌적한 피드 스크롤 경험을 제공하기 위함.

### 4. 데이터베이스 및 소스 코드 보안(Security) 강화
**문제**:
- 초기 앱 기획 단계에서 적용되었던 개방형 데이터베이스 권한(`allow read, write: if request.auth != null`)으로 인해, 악의적인 유저가 API를 호출할 경우 타인의 데이터를 임의로 수정 및 삭제할 수 있는 치명적 보안 취약점 존재.
- 앱을 외부인에게 배포(공유)할 경우 소스 코드 복제(카피캣)에 대한 방어가 확실히 인지되지 않음.

**수정 내용**:
- **Firestore Security Rules 전면 개편**: 
  - `firestore.rules`를 수정하여, 옷장(`clothes`), OOTD(`ootds`), 프로필(`users`) 등 모든 주요 컬렉션의 **수정(update) 및 삭제(delete) 권한을 데이터를 생성한 본인(userId == request.auth.uid)**에게만 부여하도록 통제.
  - 타인이 내 게시물에 좋아요를 누르거나 댓글을 달기 위해 필요한 최소한의 하위 문서(Comments 등) 생성 권한만 부분 허용.
- **소스 코드 난독화(Obfuscation) 및 Canvas 렌더링 검증**: 
  - 앱 배포 시 Flutter Web의 CanvasKit 렌더링 엔진과 Release Mode 빌드가 작동하여 소스 코드가 완벽히 난독화(Minification)됨을 확인.
  - HTML 태그가 존재하지 않아 F12 개발자 도구 등을 이용한 UI/UX 구조 무단 복제가 사실상 불가능함을 보증.

**이유**: 프라이빗 소셜 서비스로서 타인의 악의적인 데이터 훼손을 데이터베이스 레벨에서 원천 차단하고, 외부 배포 시 발생할 수 있는 소스 코드 유출 및 카피캣 위협으로부터 앱의 지식재산권을 보호하기 위함.

### 1. 가상 코디 캔버스 UI/UX 고도화 및 누끼 최적화
**문제**:
1) 등록된 옷의 누끼(배경)를 제거했을 때, 옷 주변의 빈 투명 영역(여백)이 그대로 남아있어 캔버스 상에서 옷이 너무 작게 표시되는 현상 발생.
2) 코디 캔버스에서 아이템을 추가할 때 무조건 정사각형 테두리가 생성되어, 위아래로 긴 치마나 바지의 경우 텅 빈 사각형 여백이 보여 미관상 불쾌함.
3) 모바일의 작은 화면에서 캔버스의 조그마한 옷 아이템을 정확히 두 손가락으로 집어 크기를 조절하거나 회전시키기가 매우 까다로움.

**수정 내용**:
- **누끼 여백 크롭 (Auto-Cropping)**: `web/bg_removal.js`에서 AI 배경 제거 후, 반환된 이미지의 알파(투명도) 채널을 스캔하여 실제 옷 픽셀이 있는 경계(Bounding Box)를 계산해 불필요한 투명 여백을 완벽히 잘라내도록 로직 추가.
- **비율(Aspect Ratio) 맞춤 테두리**: `lib/screens/coordination_canvas_screen.dart`의 캔버스 아이템 렌더링 컨테이너 크기를 120x120 고정에서 `BoxConstraints(maxWidth: 150, maxHeight: 150)`로 변경하여 원래 옷의 비율에 정확히 맞춰진 타이트한 검정색 테두리가 생성되도록 개편.
- **글로벌 히트박스 제스처 개편**: 캔버스의 조작 권한(제스처)을 개별 아이템 위젯에서 캔버스 배경 위젯으로 위임. 옷을 터치하여 선택만 해두면, 캔버스 배경(격자 무늬) 빈 공간 어디서든 드래그 및 핀치 줌을 통해 해당 옷의 위치/크기/회전각을 조절할 수 있는 'Global Hitbox' 시스템 구축.

**이유**: 앱의 핵심 기능인 가상 코디 캔버스에서의 사용자 경험(UX)을 대폭 개선하고, 시각적 품질과 조작 편의성을 최상의 상태로 끌어올림.
